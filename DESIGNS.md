# 超级智能体框架设计

## 0. 目标

基于 Codex 和 Claude SDK 构建一个支持长程任务运行，能在过程中自进化的 Agent Harness，支持一键之间在指定目录创建一个 Agent，并有脚本可以直接运行起来。

Codex 和 Claude 的 runtime 已经提供了原子能力：

- 文件系统
- 网络搜索
- 终端能力
- 内置工具调用
- 长时间运行的线程 / 计划 / 指令

框架应增加成长与连续性：

- 在执行过程中学习
- 持久化已学到的内容
- 将重复出现的模式提炼为可复用的技能
- 通过定期重新进入来持续推进长期任务
- 对人类和其他智能体保持可读性

## 1. 接口层

规范文件和文件夹：

- `AGENTS.md`
  稳定的行为规则。仅包含高层行为约束，具体的文件格式规范下沉到各目录的 README.md 中。
- `tool-notes/`
  部署后发现的非原生外部工具的使用说明。一个工具一个文件。
- `Memory/knowledge/`
  提炼后的长期知识。
- `Memory/episodes/`
  有界限的执行记录。
- `skills/`
  可复用的 skill 机制（由 codex/claude runtime 自动加载）。
- `mailbox/`
  人类-智能体共享邮箱。`MAILBOX.jsonl` 存放于此。
- `Runtime/`
  调度器所需的状态信息，用于安全地延续同一个 Codex 或 Claude 线程。

### 1.1 规范分层原则

`AGENTS.md` 只描述：什么时候用、什么属于这里、操作规则（创建/更新/归档条件）。

各目录的 `README.md` 描述：文件命名、frontmatter 格式、正文模板。

智能体在创建或更新文件时，必须先读取对应目录的 README.md。

## 2. Memory/knowledge（知识记忆）

### 2.1 目的

`Memory/knowledge/` 存储智能体可能再次需要的提炼后的知识。

应包含：

- 事实性知识
  术语、实体、定义、数据点和具体细节。
- 概念性知识
  模型、框架、原则、分类体系和因果结构。
- 启发性知识
  轻量级的决策规则和问题解决捷径。
- 元认知知识
  自我观察、能力边界、不确定性模式和思维触发器。

不应包含：

- 原始任务日志
- 可复用的完整工作流
- 稳定的全局行为规则
- 外部工具使用程序

这些分别属于 `Memory/episodes/`、`skills/`、`AGENTS.md` 和 `tool-notes/`。

### 2.2 目录结构

```text
Memory/
  knowledge/
    README.md          # 文件命名、frontmatter、正文模板的完整规范
    factual/
    conceptual/
    heuristic/
    metacognitive/
```

规则：

- 一个文件 = 一个可复用的知识单元
- 使用文件名和标签进行粗粒度的领域分组
- 避免过深的目录树

### 2.3 文件格式规范

命名、元数据、正文模板等详细规范存放在 `Memory/knowledge/README.md` 中。

智能体在创建或更新知识笔记时，必须先读取该 README.md。

概要：

- 命名：`<kind>--<domain>--<slug>.md`
- 必需 frontmatter：`id`, `kind`, `summary`, `last_edited_at`, `status`, `tags`
- 正文包含：核心内容、局限性、可选备注

### 2.8 操作规则

在以下情况下创建新笔记：

- 一个稳定的洞见在多个任务中反复出现
- 一个外部概念可能再次出现
- 一组易变的事实需要跟踪新鲜度
- 发现了一个稳定的盲点或决策触发器

在以下情况下更新笔记：

- 同一概念有了更好的表述
- 同一启发式规则有了更精确的边界
- 同一事实集合有了更新鲜的证据
- 同一元认知观察有了更好的校准

在以下情况下归档或删除笔记：

- 论断被证伪
- 更强的权威笔记取代了它
- 它已过时且不值得维护

## 3. Memory/episodes（执行记录）

### 3.1 目的

`Memory/episodes/` 存储有界限的执行记录。

一个执行记录是一次可评估的工作尝试的权威记录。

它支持：

- 回放
- 恢复
- 反思
- 沉淀成知识或技能

它不是：

- 完整的对话转储
- 每日日志
- 项目级日志

### 3.2 记录单元

信息源单元：

`一个执行记录 = 一个有界限的工作循环，包含一个直接目标和一个可判断的结果`

有效示例：

- 首次市场规模估算尝试
- 发现低信任来源后的重试
- 一次将证据整合为备忘录的综合过程

无效边界：

- 一条 Shell 命令
- 一个自然日
- 一个完整的多天项目

快速测试：

- 我现在试图实现什么
- 我依赖了什么上下文
- 我尝试了什么
- 产生了什么结果

如果这些问题都能回答，边界可能是正确的。

### 3.3 任务与执行记录

人类或系统可能提供一个顶层目标。
智能体将该目标分解为一个或多个任务。

定义：

- `task`（任务）
  从更广泛的目标派生出的智能体定义的执行单元。一个任务可能跨越多个执行记录。
- `episode`（执行记录）
  一个任务内的一次有界限的尝试。

规则：

- `task` 是语义分组键
- `time` 是存储键
- `episode` 是回放和学习键
- 在 V1 中，每个执行记录恰好有一个 `task_id`
- 如果一个执行记录似乎服务于多个任务，暂时不要过度建模

### 3.4 目录结构

```text
Memory/
  episodes/
    README.md          # 文件命名、frontmatter、正文模板的完整规范
    2026/
      03/
        ep--20260320T021542Z--market-research--china-ev-sizing-pass-1.md
```

规则：

- 按时间顺序存储
- 通过元数据按任务分组
- 路径稳定性比语义嵌套更重要

### 3.5 文件格式规范

命名、元数据、正文模板等详细规范存放在 `Memory/episodes/README.md` 中。

智能体在创建或更新执行记录时，必须先读取该 README.md。

概要：

- 命名：`ep--<started_at_utc>--<domain>--<slug>.md`
- 必需 frontmatter：`id`, `task_id`, `domain`, `title`, `objective`, `status`, `last_edited_at`
- 正文包含：目标、上下文快照、已采取的行动、关键证据、结果、反思、后续行动

### 3.8 生命周期规则

在以下情况下开启新的执行记录：

- 直接目标发生变化
- 完成标准发生变化
- 主要假设或策略发生实质性变化
- 上一次尝试已完成、被阻塞、失败或被放弃
- 在有意义的上下文中断后恢复工作

在以下情况下保持同一执行记录：

- 目标未变
- 成功标准未变
- 智能体仍在同一有界限的尝试中迭代

不要让一个执行记录跨越多次策略转向而保持活跃。

### 3.9 压缩

执行记录应存储与决策相关的信息，而非原始输出。

应包含：

- 决定性的观察
- 重要的命令、文件或 URL
- 矛盾的证据
- 失败点
- 推理变化

默认应避免：

- 完整的终端输出转储
- 每一个中间想法
- 每一个网页摘录
- 重复复制的上下文

学习图谱：

`task -> episode -> reflection -> knowledge / skill`
`任务 -> 执行记录 -> 反思 -> 知识 / 技能`

## 4. skills（技能）

### 4.1 目的

`skills/` 存储智能体可以有意识地加载和运行的可复用能力。

技能如何编写不在此处规定。技能由 codex/claude runtime 自动加载，无需在 AGENTS.md 中额外说明。

### 4.2 边界

使用以下规则：

- `AGENTS.md`
  始终生效的行为约束和工作风格。
- `tool-notes/`
  如何使用非原生外部工具。
- `Memory/knowledge/`
  事实、概念、启发式规则和元认知笔记。
- `Memory/episodes/`
  有界限的尝试中发生了什么。
- `skills/`
  按需加载的可复用可执行能力。

## 5. tool-notes/（工具文档）

### 5.1 目的

`tool-notes/` 记录部署后遇到的非原生外部工具的使用方法。一个工具一个文件。

它不解释原生 Codex/Claude 能力，例如：

- 文件系统访问
- 网络搜索
- 终端访问
- 内置工具调用

### 5.2 什么属于这里

在以下情况下在 `tool-notes/` 中创建工具笔记：

- 它不是原生运行时的一部分
- 智能体后来才获得了访问权限
- 其用法不够直观，遗忘会导致摩擦
- 它可能在多个任务中反复出现

示例：

- 内部 REST API
- 通过适配器暴露的第三方 SaaS 管理控制台
- 自定义 MCP 服务器
- 为特定工作流安装的本地 CLI

不要记录在此处：

- 通用业务方法论
- 技能程序
- 使用工具时学到的事实
- 原生 Codex/Claude 工具行为

### 5.3 文件格式规范

命名、元数据、正文模板等详细规范存放在 `tool-notes/README.md` 中。

智能体在创建或更新工具笔记时，必须先读取该 README.md。

概要：

- 命名：`tool--<slug>.md`
- 必需 frontmatter：`id`, `name`, `last_edited_at`, `status`
- 正文包含：用途、访问方式、输入、输出、认证/配置、调用模板、失败模式

## 6. Heartbeat（心跳机制）

### 6.1 目的

心跳是框架层的编排机制，用于定期决定是否继续在同一 Codex或Claude 线程上工作。

默认间隔：每 20 分钟。

### 6.2 循环

每次心跳唤醒时：

1. 先从 `mailbox/MAILBOX.jsonl` 读取新消息
2. 如果 `awaiting_human = true` 且没有新的人类消息，结束本次运行
3. 如果没别的情况，则继续之前的对话运行

## 7. mailbox/（邮箱 / 询问人类）

### 7.1 目的

`mailbox/MAILBOX.jsonl` 是人类-智能体之间唯一的通信界面。

用途：

- 智能体进度更新
- 智能体请求人类帮助
- 人类回复、决策和指令

### 7.2 文件格式规范

消息格式、操作规则等详细规范存放在 `mailbox/README.md` 中。

智能体在写入邮箱前，必须先读取该 README.md。

概要：

- 每行是一个 JSON 对象
- 必需字段：`id`, `ts`, `role`（`agent | human`）, `task_id`, `message`
- 仅追加；不修改旧消息
- 智能体必须在每次唤醒时读取新的邮箱条目
- 如果智能体正在等待人类输入且没有新的人类消息，不应继续被阻塞的任务
